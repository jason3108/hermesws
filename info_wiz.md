## 1. Securing the AI Edge: Wiz and Cloudflare Integrate for End-to-End AI Protection

**发布日期**：2026-04-14  
**作者**：Bandhna Bedi, Guy Weiss, Gilad Hoze  
**原文链接**：[Wiz and Cloudflare AI Security Integration](https://www.wiz.io/blog/wiz-cloudflare-ai-security-integration)  
**本地PDF**：[blog/wiz-cloudflare-ai-security-integration.pdf](blog/wiz-cloudflare-ai-security-integration.pdf)

### 中文摘要

**背景与挑战**

安全行业多年来一直致力于将安全左移到开发流程更早的阶段，通过嵌入安全防护、在开发生命周期早期集成检查等方式来提升安全性。然而，AI的崛起打破了这一进程。借助AI，开发者现在可以在几分钟内通过自然语言提示快速构建复杂的AI应用，这些应用由模型和AI代理驱动，能够连接敏感数据、调用内部API并在云环境中运行。在这种前所未有的创新速度下，安全团队已经无法跟上节奏。AI不仅加速了开发过程，更重要的是，它将开发者的开发旅程与安全团队的保障旅程彻底解耦了。

与此同时，获取AI使用情况的可见性本身就变得异常困难。传统的监控手段（如通过日志追踪已知AI提供商）只能讲述故事的一部分。根据Wiz Research的数据，实际上88%的组织运行着自托管AI模型，79%的AI工作负载能够接触敏感API或数据而没有任何安全防护。影子AI、恶意代理和公开暴露的端点正在以前所未有的速度扩大攻击面。

**Wiz与Cloudflare的整合方案**

这正是Wiz与Cloudflare整合的核心价值所在——将深度云环境上下文与边缘级别的可见性及AI端点保护相融合，使团队能够全面了解AI应用的构建方式、暴露情况以及如何保护它们。

具体而言，Wiz提供深度云上下文，能够将AI工作负载映射到为其提供动力的底层基础设施、身份、数据存储和代码。它能够识别哪些AI模型是自托管的、哪些服务与之交互，以及敏感数据或配置错误可能在何处引入风险——为团队提供清晰的环境AI足迹视图，包括哪些应用端点在安全图谱上处于暴露状态。

**覆盖整个AI应用生命周期的保护**

但AI应用并非孤立存在——它们通过DNS提供服务，位于负载均衡器之后，并连接到面向公众的基础设施。这就是Cloudflare整合的另一个关键价值：它还为团队提供通过Cloudflare暴露的DNS区域、记录和云工作负载的可见性，使他们能够在边缘层面看到AI服务是如何暴露的，并检测这些配置中的敏感数据或密钥。

当AI端点通过配置错误的DNS记录或未受保护的负载均衡器可访问时，这不仅仅是基础设施问题，更是AI安全问题。借助Cloudflare AI Security for Apps整合，Wiz的AI可见性进一步扩展，显示哪些AI应用端点受到Cloudflare保护，哪些仍需要保护。当用户、API或其他代理访问这些端点时，Cloudflare的安全防护会强制执行保护措施，抵御提示注入和PII暴露等威胁。

通过连接这些层面，安全团队获得了统一的AI攻击面视图，从构建AI应用的云环境一直到暴露这些应用的面向公众的端点。因此，当在Wiz中识别出带有提示注入漏洞的暴露AI模型时，团队可以通过部署Cloudflare AI Security for Apps来采取行动，防止潜在的滥用或数据泄露。

**总结**

AI是近一代人中最具变革性的技术，它推动了无数能力的发展。但对于大多数企业来说，它可能是一个黑箱。在与当今的CISO交流时，他们正在努力平衡两件事：一方面是成为AI创新的推动者，另一方面是对抗在整个组织中不受控制的影子AI，因为传统安全工具在这一层面基本上毫无用处。Cloudflare与Wiz的合作有助于解决这一权衡问题。现在，他们提供了一种解决方案，允许在不影响数据安全的前提下快速进行AI创新。

通过将Wiz的端到端可见性与Cloudflare的边缘保护相结合，该合作伙伴关系弥合了AI风险管理中的一个关键空白。该解决方案为组织提供了AI应用端点和共享风险上下文的统一视图，帮助他们在威胁（如提示注入和影子AI）发生之前就加以阻止。借助统一视图，安全团队可以全面了解AI攻击面，从构建AI应用的云环境一直到暴露这些应用的公共端点，从而使他们能够在风险成为问题之前采取行动。

## 2. Introducing Shadow Data Detection: Reduce Cost and Risk Across Your Cloud

**发布日期**：2026-04-14  
**作者**：Snegha Ramnarayanan, Alma Raziel, Matika Lidgi  
**原文链接**：[Introducing Shadow Data Detection in Wiz](https://www.wiz.io/blog/introducing-shadow-data-detection-in-wiz)  
**本地PDF**：[blog/introducing-shadow-data-detection-in-wiz.pdf](blog/introducing-shadow-data-detection-in-wiz.pdf)

### 中文摘要

**AI改变了数据保留方式**

AI不仅仅改变了我们的构建方式——它改变了我们保留数据的方式。在云环境中，数据不断被创建、复制、版本化和保留。存储可以轻松扩展，但数据所有权却无法同步跟上。随着时间的推移，累积的不仅仅是需要保护敏感数据——还有那些已经不再有用途的数据。陈旧的对象、重复的文件、过度保留的版本。这些都是影子数据。虽然这些数据可能并非总是敏感的，但它仍然很重要——因为每个不必要的对象都会增加成本和暴露面。

**影子数据检测功能发布**

Wiz今天发布了影子数据检测功能——这是一项新能力，帮助组织识别并消除其云环境中管理不善的数据，从而降低存储成本和不必要的风险。

理解敏感数据仍然是任何强大数据安全计划的基础。组织需要知道哪些数据重要，它们存储在哪里，谁可以访问它们。影子数据检测建立在这一基础之上。即使对敏感数据有强大的可见性，很大一部分数据足迹通常也管理不善——重复、陈旧或根本不必要。无论这些数据是否敏感，它们都会增加云成本和攻击面。

借助影子数据检测，Wiz将DSPM（数据安全态势管理）从识别敏感数据扩展到理解哪些数据是低效的、未使用的或过度保留的——以便团队能够采取行动。在早期环境中，Wiz已经在客户的云存储桶中识别了超过1艾字节（1 exabyte）的数据，其中很大一部分被归类为冗余或陈旧——直接转化为有意义的成本节约和减少的暴露面。在Wiz自己的环境中，这发现了存储桶中TB级的冗余数据，进一步印证了减少不必要支出和暴露的机会。

结果是清晰的可见性——知道要保留什么、要优化什么、要删除什么——以及对成本和风险的直接影响。

**影子数据检测的实际应用**

影子数据检测作为Wiz平台的一部分工作，使用已经部署在各云环境中的相同连接。Wiz分析由云提供商直接生成的数据清单报告，以评估数据是如何存储和管理的，而不影响性能。在此基础上，它识别重复对象、过度版本控制和陈旧数据等模式，将其作为数据发现（Data Findings）呈现，并带有清晰、可操作的上下文——同时使用内置检测逻辑和团队根据自身数据管理策略定义自定义规则的能力。

这些洞察集中在一个专用的影子数据仪表板中，提供跨环境低效数据的集中视图——包括存储影响、潜在成本节约和不必要的暴露领域。从高层视图，团队可以快速识别最大的机会在哪里，哪些环境或资源驱动最多的低效。发现结果根据影响进行组织和优先级排序，让团队清楚了解其环境中最关键的低效问题。然后团队可以深入研究特定发现，了解到底是什么导致了低效，应该采取什么行动。这将数据泛滥转化为团队可以衡量、优先级排序和修复的问题——减少数据直接减少成本和暴露。

**跨团队协作解决问题**

影子数据不属于单一团队，这正是它成为问题的原因。安全团队希望减少不必要的暴露。平台团队希望提高效率和成本优化。数据团队希望加强治理和生命周期控制。影子数据检测将这些问题汇集在一起，提供一致共享的视图，以便团队能够就低效问题所在、优先级以及如何采取行动达成共识。由于这是Wiz平台的一部分，这些洞察直接连接到更广泛的上下文——包括访问、配置和风险——因此团队不仅了解数据存在的位置，还了解它如何导致真正的暴露。

## 3. Claude Mythos: Preparing for a World Where AI Finds and Exploits Vulnerabilities Faster Than Ever

**发布日期**：2026-04-10  
**作者**：Ami Luttwak  
**原文链接**：[Claude Mythos: What You Need to Know](https://www.wiz.io/blog/claude-mythos)  
**本地PDF**：[blog/claude-mythos.pdf](blog/claude-mythos.pdf)

### 中文摘要

**Claude Mythos是什么？**

Claude Mythos是Anthropic开发的一款新的、尚未公开发布的前沿模型。该模型能够自主发现主流操作系统和Web浏览器中的数千个零日漏洞。目前已公开发布的前沿模型在一定程度上已经能够进行补丁diff分析和生成漏洞利用代码——当给定相关代码片段时。然而，Anthropic还展示了Mythos能够接收CVE标识符和git提交哈希作为输入，然后在数小时内以相对较低的成本自主生成完整的有效漏洞利用代码。此外，据报道它还能将多个漏洞链接在一起，并逆向工程闭源二进制文件。这一发布标志着一种趋势的延续（这种趋势在过去一年中已经加速）：AI模型开始擅长度量漏洞研究技术，考虑到其核心的正向反馈循环，这完全合乎逻辑。

**近期应该期待什么？**

考虑到当前行业发展轨迹，我们需要审视安全产品的构建方式和防御者的运作方式发生的变化。这些新兴能力应该被纳入我们安全计划和产品的核心，以便我们能够适应一个 eventual的现实——恶意行为者将用他们自己的工具做同样的事情。目前我们正在从三个主要阶段的时间维度来思考AI赋能的安全研究问题空间。我们相信，每个阶段的成功都需要安全团队和安全厂商采取新的策略和工具：

**短期：期待更多CVE**

目前，Mythos只在负责任的行为者手中——微软、谷歌和Linux基金会等关键软件基础设施提供商。该模型尚未公开发布，Anthropic表示他们没有计划改变这一状况。因此，最直接的后果就是更多的CVE。使用Mythos等模型的安全研究人员将发现零日漏洞，证明其可利用性，并负责任地向软件供应商和开源项目维护者披露。每个使用这些模型发现的漏洞最终都会（或至少有望）成为带有补丁的公开CVE。这些CVE不太可能立即附带公开的漏洞利用代码，但攻击者将投入资源进行（AI辅助的）补丁diff分析，无论受影响的产品是开源还是闭源的。最终，这将导致更多最近发布的漏洞在野外被并行利用。除非安全团队扩大自己的响应能力，否则攻击者将有更多机会利用未打补丁的系统。

**中期：为"Y2K时刻"做准备**

这只是时间问题——具有与Mythos类似能力的模型（来自Anthropic、谷歌、OpenAI、DeepSeek、阿里巴巴等）终将对公众开放。我们估计，在大约12-18个月内，这些能力将渗透到任何人都可以在本地运行且不受限制的开源模型中。从那时起，我们应该假设恶意行为者将能够使用AI模型大规模地发现和武器化零日漏洞，并在漏洞公开披露后数小时内迅速武器化n日漏洞。

除了安装型软件中的漏洞外，大多数现代企业当今的主要攻击面是他们自己的API端点和Web应用程序，这些漏洞往往是逻辑驱动的：身份验证绕过、授权中断、暴露的端点和配置错误的访问控制。事实上，目前的前沿模型已经在识别和利用这些弱点方面表现出色。

然而我们保持乐观。正如历史所表明的，cyber领域中每一种可以被攻击者使用的新技术也可以被防御者使用，最终结果将是更安全的系统。内存模糊测试器（历史上被漏洞研究人员使用）现在已嵌入核心开发流程，最终产生更安全的软件。

## 4. Cloud Threats Retrospective 2026: What AI Changed (and What It Didn't)

**发布日期**：2026-04-09  
**作者**：Wiz Threat Research  
**原文链接**：[Cloud Threats Retrospective 2026](https://www.wiz.io/blog/cloud-threat-retrospective-2026)  
**本地PDF**：[blog/cloud-threat-retrospective-2026.pdf](blog/cloud-threat-retrospective-2026.pdf)

### 中文摘要

**经过验证的云风险路径依然重要**

在最新的云威胁回顾报告中，Wiz分析团队结合公开记录的云事件、云遥测数据和实际调查发现：许多推动攻击者活动的风险依然熟悉，尽管周围的环境和条件发生了变化。

在2025年分析的大多数云事件中，初始访问主要来自于众所周知的弱点，包括漏洞、暴露的密钥和配置错误。这些入口点并非新颖，但它们仍然非常有效，约占记录在案的云入侵事件的80%。发生变化的不是这些风险的存在，而是它们出现的环境以及被利用的速度。

**系统性弱点推动更广泛影响**

2025年一些最具影响的事件表明，系统性弱点如何能够放大影响，远远超出单一环境的范围。当攻击者通过共享基础设施、受信任的集成或广泛使用的组件获得访问权限时，一个弱点可能导致许多组织遭受连锁影响。这些事件强化了一个重要转变：理解云风险现在需要超越单个资产，审视连接它们的依赖关系和关联。

**AI扩展了熟悉风险的分布范围**

AI似乎并没有在2025年引入全新的云风险类别，但它以有意义的方式扩展了云攻击面。新的AI服务、管道、身份和数据路径增加了熟悉问题（如配置错误或暴露的凭证）可能出现的位置，通常更接近敏感数据和高价值工作负载。随着AI采用的加速，许多组织发现他们管理新组件的速度超过了安全实践能够充分适应的速度。

**AI支持现有攻击者工作流程**

在Wiz研究团队分析的事件中，AI最常被观察到是支持和加速现有攻击者行为，如侦察、自动化和访问后活动。这些能力减少了入侵某些阶段中的摩擦和努力，但它们主要建立在防御者已经认可的技战术之上。

**这对云安全团队意味着什么**

2025年的收获并非一切照旧。相反，当熟悉的风险与规模、共享信任和AI驱动环境相结合时，可能导致戏剧性的安全结果。能够保持对暴露面、身份以及风险如何在云、开发系统和AI系统间传播的可见性的安全团队，更有能力在攻击活动升级之前检测和中断它。

## 5. Bringing Security Visibility to Vercel with Wiz

**发布日期**：2026-04-09  
**作者**：Mike McGuire, Bandhna Bedi, Gilad Hoze  
**原文链接**：[Wiz Vercel Integration](https://www.wiz.io/blog/introducing-wiz-vercel-integration)  
**本地PDF**：[blog/introducing-wiz-vercel-integration.pdf](blog/introducing-wiz-vercel-integration.pdf)

### 中文摘要

**现代应用开发的变革**

现代应用开发已经发生了巨大变化。如今的应用越来越使用前端框架、无服务器功能和边缘基础设施构建。Vercel等平台使开发者能够在几秒钟内部署全球分布式应用，而无需管理传统基础设施。这种转变释放了惊人的速度——开发者可以以前所未有的速度迭代，通过基于Git的工作流和预览环境进行更新，每次提交都会自动启动预览环境。但这种速度给安全团队带来了新的挑战。当底层基础设施被抽象化时，可见性往往会随之消失。安全团队可能不知道部署了哪些项目、哪些域名暴露在外，或者访问配置是怎样的。简而言之，如果你不管理基础设施，你如何保护它？

**Wiz Vercel集成：将部署与云上下文连接**

Wiz Vercel集成的核心价值在于将Vercel环境纳入与云其余部分相同的安全上下文。 Wiz从Vercel获取资源（如项目、域名、团队成员和防火墙配置），并将其纳入Wiz安全图谱（Security Graph）。这创建了持续更新的视图，不仅能显示什么配置错误，还能显示它如何影响整个系统。

借助这种共享上下文，安全团队可以追踪发生的变化，看到新部署何时变得公开可访问，识别配置错误（如过于宽松的防火墙规则或不完整的机器人保护），并理解这些问题引入了哪些攻击路径，以便更好地进行风险优先级排序。开发者则可以看到他们拥有的应用程序的暴露情况和潜在影响，使他们能够直接采取行动——调整访问权限、修复配置错误或解决暴露——而不必减慢工作流程。通过将安全民主化并授权开发者拥有他们构建的应用程序的安全性，安全团队可以在不失去可见性的情况下快速行动，确保在应用程序构建和发布时，它们也得到持续的理解和保护。

**与开发者速度同步的安全**

Vercel上的开发者行动迅速，通过Git工作流在几秒钟内迭代功能并部署全球分布式应用程序。借助Wiz + Vercel集成，组织可以获得对抽象资产的可见性、可操作的优先级风险情报，以及开发速度与安全信心之间的协作。主动安全控制——包括云配置规则、图谱控制和攻击面策略——允许团队自动执行最佳实践。开发者可以快速行动，知道防护措施会及早捕获有风险的配置，而安全团队则与快速移动的工作流程保持同步。这种方法确保安全和开发同步运作：开发者了解他们工作的真实世界影响，安全团队可以基于优先级风险情报采取行动，实现安全的高速开发。

## 6. Primer on GitHub Actions Security - Threat Model, Attacks and Defenses (Part 1/2)

### 中文摘要

**GitHub Actions 的安全威胁背景**

GitHub Actions 是最受欢迎的 CI/CD 平台之一，它使开发者能够通过自动化工作流来构建、测试和部署代码。然而，这种便利性也引入了新的攻击面。GitHub Actions 基于 YAML 配置文件，运行在 GitHub 托管的基础设施上，允许第三方提交工作流，这使得攻击者可以利用这些工作流来执行恶意代码。

**已知攻击向量**

Wiz Research 识别了多种针对 GitHub Actions 的攻击向量，包括：使用来自不受信任来源的第三方 Actions、暴露的 GitHub Tokens、恶意的工作流注入、以及通过 pull_request_target 事件触发的攻击。这些攻击向量已被多个威胁行为者（如 TeamPip 和 prt-scan）在真实的供应链攻击中利用。

**防御策略**

文章提供了详细的防御指南，包括：限制 Actions 的权限范围、使用 OpenID Connect (OIDC) 进行身份验证、避免使用不受信任的第三方 Actions、实施工作流签名验证、以及监控异常行为。

## 8. The Agile FedRAMP Playbook, Part 4: Reactive Risk Management through Enriched Incident Response

**发布日期**：2026-03-06

### 中文摘要

**FedRAMP 合规与事件响应融合**

FedRAMP（Federal Risk and Authorization Management Program）是美国联邦政府云安全授权的标准框架。随着越来越多的联邦机构采用云服务，FedRAMP 合规已成为云服务提供商进入政府市场的必要条件。然而，合规并不等于安全——组织需要在满足合规要求的同时，建立强大的事件响应能力。

**Enriched Incident Response 概念**

Wiz 在本文中提出了"Enriched Incident Response"的概念，即在事件响应过程中融合云环境的多维度上下文。传统的 IR 流程往往缺乏足够的上下文，导致响应速度和准确性受限。而 Enriched IR 通过将运行时活动、云配置、身份数据和攻击路径信息相融合，使安全团队能够在事件发生时做出更快、更准确的决策。

**实践指南**

本文提供了具体的实施步骤，包括：建立云资产的完整清单、实现身份与访问的实时监控、部署自动化响应工作流、以及与 FedRAMP 合规框架保持一致。
